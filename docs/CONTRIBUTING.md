## 코드 리뷰 규칙

* 모든 PR은 작성자를 제외한 팀원에게 리뷰를 요청한다.
* 리뷰어는 PR의 변경 내용이 연결된 Issue의 작업 범위와 일치하는지 확인한다.
* 리뷰어는 코드 또는 문서가 팀의 브랜치 규칙, 커밋 메시지 규칙, PR 작성 규칙을 따르는지 확인한다.
* 리뷰 코멘트는 `LGTM`, `좋아요`, `확인했습니다`만 단독으로 작성하지 않는다.
* 리뷰어는 파일 또는 라인을 기준으로 질문, 대안 제안, 리스크 지적, 개선 제안 중 1개 이상을 작성한다.
* PR 작성자는 리뷰 코멘트에 최소 1회 이상 답글을 남기거나, 수정 커밋을 통해 리뷰 내용을 반영한다.
* 리뷰를 반영한 경우 어떤 부분을 수정했는지 PR 코멘트로 간단히 남긴다.
* 리뷰 반영 내역과 관련 PR 링크는 최종 제출 시 `SUBMISSION.md`에 정리한다.

### 리뷰어 확인 항목

* 변경 내용이 Issue의 목적과 일치하는가?
* 불필요한 파일이나 작업 범위 밖의 변경이 포함되지 않았는가?
* 함수명, 변수명, 파일명이 역할을 이해하기 쉽게 작성되었는가?
* 예외 상황이나 빈 값 처리 기준이 필요한 경우 명확히 작성되었는가?
* 문서 변경의 경우 문장이 이해하기 쉽고 팀 규칙으로 사용하기에 충분한가?
* PR 본문에 변경 사항, 변경 이유, 검증 방법이 작성되어 있는가?

### 리뷰 코멘트 예시

```text
src/data_utils.py의 평균 계산 함수에서 빈 리스트가 들어오는 경우의 처리 기준이 필요해 보입니다.
빈 리스트 입력 시 0을 반환할지, ValueError를 발생시킬지 docstring에 명시하면 좋겠습니다.
```

```text
docs/CONTRIBUTING.md의 코드 리뷰 규칙이 명확합니다.
다만 리뷰 반영 내역을 SUBMISSION.md에 정리한다는 기준을 추가하면 제출 증빙 관리에 더 도움이 될 것 같습니다.
```

### PR 작성자 응답 예시

```text
좋은 의견 감사합니다. 빈 리스트 입력 시 ValueError가 발생하도록 처리하고, docstring에 예외 조건을 추가했습니다.
```
