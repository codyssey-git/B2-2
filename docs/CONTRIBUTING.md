# 협업 가이드

팀 과제 시작 후 협업 규칙을 이곳에 정리합니다.

## 브랜치 네이밍 규칙

브랜치명은 **작업 유형/이슈 번호** 형식으로 작성한다.

```bash
작업유형/이슈번호
```

### 예시

```bash
docs/2
feat/3
fix/4
refactor/5
chore/6
```

### 작업 유형

| 유형       | 설명                                 |
| ---------- | ------------------------------------ |
| `feat`     | 새로운 기능 추가                     |
| `fix`      | 버그 수정                            |
| `docs`     | 문서 작성 및 수정                    |
| `refactor` | 기능 변경 없는 코드 구조 개선        |
| `chore`    | 설정, 빌드, 패키지 관리 등 기타 작업 |

### 작성 규칙

- 브랜치명은 소문자로 작성한다.
- 공백은 사용하지 않는다.
- 이슈 번호 앞에는 `#`을 붙이지 않는다.
- `/`를 기준으로 작업 유형과 이슈 번호를 구분한다.
- 하나의 브랜치는 하나의 이슈를 기준으로 생성한다.

## PR 작성 규칙
* 모든 작업은 Issue 생성 후 진행한다.
* 작업 완료 후 `feature/*` 브랜치에서 PR을 생성한다.
* 하나의 PR에는 하나의 작업만 포함한다.
* 관련 없는 파일 수정은 포함하지 않는다.
* `main` 브랜치로 직접 push하지 않는다.

## 목적

PR(Pull Request)은 작업 내용을 팀원에게 공유하고 코드 리뷰를 받기 위한 절차이다. 모든 변경 사항은 PR을 통해 `main` 브랜치에 반영한다.


## PR 제목 규칙

PR 제목은 커밋 메시지 컨벤션과 동일한 형식을 사용한다.

예시:

```text
feat: 문자열 길이 검증 함수 추가
docs: PR 작성 규칙 작성
fix: 길이 검증 함수 예외 처리 수정
```

## PR 본문 필수 항목

모든 PR 본문에는 아래 내용을 포함해야 한다.

### 연결 이슈

```text
Closes #이슈번호
```

### 변경 사항 (What)

* 어떤 내용을 추가하거나 수정했는지 작성

### 변경 이유 (Why)

* 해당 변경이 필요한 이유 작성

### 테스트/검증 방법 (How)

* 변경 사항을 어떻게 확인했는지 작성

예시:

```text
## 연결 이슈
Closes #7

## 변경 사항
- validate_length 함수 추가

## 변경 이유
- 문자열 길이 검증 기능 제공

## 테스트/검증 방법
- 최소 길이, 최대 길이 조건 테스트 완료
```

## 리뷰 반영 규칙

* 리뷰 의견을 확인한 후 답글을 작성한다.
* 필요한 경우 수정 커밋을 추가한다.
* 리뷰 반영 후 변경 내용을 PR에 남긴다.
* 최소 1명 이상의 승인 후 병합한다.

## 병합 규칙

* 승인 없이 병합하지 않는다.
* 충돌 발생 시 해결 후 다시 리뷰를 요청한다.
* 병합 후 연결된 Issue가 정상적으로 종료되었는지 확인한다.
